## Key design decisions

- **Custom EDF binary parser for headers**: EDF headers are read using a hand-written binary parser rather than pyedflib. pyedflib was insufficiently robust for the heterogeneous EDF files encountered in real-world datasets (encoding edge cases, non-standard headers). This parser is used **only for header inspection** — it never loads signal data.
- **Sampling frequency = samples_per_record / duration_data_record**: the per-channel 8-byte header field is the *number of samples per data record*, not the sampling frequency. They coincide only when `duration_data_record == 1 s` (classic EDF); EDF+ files often use another record duration (e.g. 0.1 s or 2 s), so the rate must be divided by `duration_data_record` to match MNE (e.g. `40 / 0.1 = 400 Hz`, `512 / 2 = 256 Hz`). The custom parser stores the raw field as `samples_per_record` and derives `sampling_frequency` from it. Keep this in sync across all tools sharing the parser (`1_inspect_edf*`, `2_select&remap_channels_edf*`). See SPEC.md for details.
- **MNE for signal loading**: Whenever actual signal data needs to be loaded (e.g. for spectral analysis or future preprocessing), `mne.io.read_raw_edf()` is used. Do not use pyedflib for signal loading.
- **edfio for EDF export**: When writing EDF files (e.g. in `generate_test_data.py`), use `edfio.EdfSignal()` and `edfio.Edf()` directly — not `mne.export.export_raw()`. Direct edfio usage allows per-channel physical range overrides that MNE's export layer does not expose.
- **In-place header anonymization (`1bis_anonymize_edf*`)**: To anonymize without re-exporting (slow), the anonymizer writes a **new** file (`shutil.copy2`) then overwrites **only** the fixed-width identity fields of the 256-byte general header: `patient_id` (bytes 8–88) and `recording_id` (bytes 88–168). Everything from byte 256 onward (per-channel headers + all signal records) is byte-for-byte identical — verified by `sha256(file[256:])` before/after plus an equal-file-size check. **Originals are never modified**; the user is told to delete them after verifying. `patient_id` → `X X 30-DEC-1899 X_X` (Compumedics anonymized format — strips name *and* birthdate, keeps a placeholder date). `recording_id` keeps the real `Startdate <dd-MMM-yyyy>` token (the recording night is **not** direct PII and downstream sleep tools need it) and blanks only the trailing admin/technician/equipment fields to `X X X`. **`start_date`/`start_time` (bytes 168–184) are left untouched** — verified against real files, this is exactly what Compumedics anonymized exports do (only the birthdate inside `patient_id` is replaced). Companion files (hypnogram `.txt`, `.edf.XML`, `*_event_xml.csv`) are copied + renamed to the new stem with **content unchanged** — verified that Compumedics companions hold no patient name (the `.edf.XML` is a `CMPStudyConfig` of scoring/montage settings); a warning still tells the user to re-check their own export profile.
- **Dual delivery (Jupyter + Voila)**: Every user-facing tool exists in both forms. The Voila version hides all code cells; the Jupyter version is kept in sync for debugging. Always maintain both versions when modifying a tool.
- **TSV + HTML outputs**: TSV files are machine-readable for scripting; HTML reports are human-readable for non-programmers.
- **MNE duplicate channel suffixes**: MNE ≥ 1.8 renames duplicate channel names by appending `-0`, `-1`, etc. Use `drop_suffix_duplicates()` to keep only the `-0` variants, and `adapt_remap_dict_to_suffixes()` to match base channel names in remap dicts to MNE's suffixed names. Apply this pattern whenever loading an EDF with MNE and using a channel remap dict.
- **Physical bounds from MNE internals**: MNE 1.9 stores EDF physical range as `physical_max + offset` in `_raw_extras`, not as explicit `physical_min`/`physical_max`. Use `get_phys_bounds_uV()` (defined in `5_quality_overview_voila.ipynb`) to reconstruct these bounds when needed.
- **EOG/EMG/ECG channel detection**: classify non-EEG channels by **transducer type OR channel name** (method from `inspect_edf_voila_EMG.ipynb`): EOG = transducer `EOG` or name `eog`; ECG = transducer `ECG`/`EKG` or name `ecg`/`ekg`; EMG = transducer `EMG` or name `emg`/`chin`/`menton` (the `chin|menton` additions cover Compumedics chin-EMG labels). EEG uses `KNOWN_EEG_CHANNEL_RE` (full 10-10 + mastoids + literal `EEG`) or transducer `EEG`/`AGAGCL ELECTRODE`, excluding anything already matched as EOG/ECG/EMG. Reuse `detect_channel_types()` (in `7_live_explore_1file.ipynb`) when a tool needs the scoring montage, and pre-fill it as an editable selection so the user can correct misses.
- **Compumedics event annotations**: the `*_event_xml.csv` companion next to an EDF is already a parsed table (`Name, Start, Duration` in seconds: arousals, apnea/hypopnea, limb movement, SpO2 desaturation…). Read this CSV directly (no XML parsing) to overlay events; `7_live_explore_1file.ipynb` does this via `load_events_csv()`.
- **Event sourcing & harmonization (`4_remap_events_edf*`)**: scored events are read **CSV-first / XML-fallback** via a shared `load_events(edf_path)` helper — the Compumedics `*_event_xml.csv` (`Name, Start, Duration`) is tried first, then the `<ScoredEvents>` of the `*.edf.XML` (Profusion `CMPStudyConfig`); both verified equivalent on ICEBERG (the tool ships an optional CSV-vs-XML consistency check writing `event_source_mismatch.tsv`). Event *configurations* group files by their `frozenset` of **unique event labels actually present** (not the `<ScoredEventSettings>` profile catalogue). Harmonization produces a single **global** flat JSON `config_param/event_remap.json` (`{raw_label: canonical_label}`, `null` = ignore), merged incrementally across runs with the lenient JSON loader and seeded by `DEFAULT_EVENT_MAPPING` (snake_case canonical vocabulary; arousal subtypes kept, limb laterality collapsed). **TODO**: back-port the XML fallback into `live_explore`'s `load_events_csv()` (see `tools/TODO_live_explore_event_xml_fallback.md`).

## Development context

- **Developer background**: The primary developer is a sleep research engineer with domain expertise in PSG/EEG but limited software engineering experience. Prioritize readability and correctness over abstraction or cleverness.
- **No unnecessary abstraction**: Keep functions explicit and readable. Three clear lines beat a clever one-liner. Do not introduce helper functions or classes unless the complexity clearly justifies it.
- **Error handling in notebooks and new features**: When implementing a new feature or creating a notebook, add `try/except` blocks proactively. Distinguish fatal vs non-fatal failures: a fatal step (e.g. file I/O, epoching) should add the item to a `failed` list and `continue` to the next iteration; a non-fatal step (e.g. re-referencing, report generation) should print a `⚠` warning and continue processing. In Voila notebooks specifically, every button callback and every per-item processing loop must be wrapped so that a single failure never silently crashes the whole run or freezes the UI. Always surface error messages to the user via a widget (e.g. `HTML`, `Output`) or `print()` inside a `widgets.Output` context.
- **Skip existing reports + robust global output updates**: Every processing tool that generates one output file per participant must include a "Skip participants with an existing report" checkbox (checked by default). The pattern has three parts: (1) when the data folder or output folder is selected, an `existing_reports_info` HTML widget displays how many participants already have a report; (2) when "Load participants" is clicked, participants with an existing report are filtered out of the list (not shown), and the info message reports how many were hidden; (3) in the run loop, `attempted_ids` (a `set`) is populated just after the skip check — only participants actually entered into the processing `try:` block are added. Global output files must then be updated using `attempted_ids` as follows: cumulative per-row files (e.g. `global_epoch_rejection.tsv`) are merged with the existing file by removing rows where `file_id ∈ attempted_ids` then concatenating new rows; aggregated summary files (e.g. `global_rejection_by_stage.tsv`) are fully regenerated from all individual per-participant files present in the output folder (not just the current run); `failed.tsv` uses the same merge logic, and entries for participants that previously failed but now succeed are automatically removed (file deleted if it becomes empty). Reference implementations: `5_quality_overview_voila.ipynb` and `6_preprocessing_voila.ipynb`.
- **Comments**: Add them only where the EEG/PSG domain logic would not be obvious to a reader unfamiliar with sleep research (e.g. why a 500 µV threshold flags clipping, or why stage 4 is remapped to N3).
- **No formal test suite**: Validate tools against real EDF files from the dataset. Do not add pytest or unittest infrastructure unless explicitly requested.
- **OS**: Windows 10 with Conda. PowerShell is the default shell. Use forward slashes in Python path strings (MNE and pathlib handle them correctly on Windows).
- **Normalize path comparisons**: whenever two filesystem paths are compared as strings (equality, `in`, `.isin()`, set membership, dict-key lookup), wrap both sides in `os.path.normcase(...)` first. On Windows this removes drive-letter case (`C:` vs `c:`) and `/` vs `\` mismatches (a no-op on POSIX, so it stays cross-platform). Keep the stored/displayed path in its original form; normalize only at the comparison. This prevents "already-processed / skip" checks from silently failing when one path comes from disk and the other from a fresh folder scan. Applied across all current tools (audit complete); see *Cross-cutting procedures → Path comparison normalization* in SPEC.md.
- **Running Python scripts from the terminal**: `conda` is NOT in the PATH. miniforge3 is installed at `$env:LOCALAPPDATA\miniforge3` (not `$env:USERPROFILE\miniforge3`). To run a script, always use the explicit executable path:
  ```powershell
  & "$env:LOCALAPPDATA\miniforge3\envs\inspect_edf\python.exe" tools/my_script.py
  ```
  Do NOT use `conda run`, `conda activate`, or bare `python` — they will all fail.
- **Editing large notebooks (rename-to-`.txt` method)**: The standard notebook tools fail on the larger tool notebooks because `Read` renders **all** cells and ignores `offset`/`limit` on `.ipynb`, so any notebook whose *total* size exceeds the ~25 k-token `Read` cap cannot be read (this is driven by total file size, not by the single-mega-cell structure — the multi-cell Jupyter twins hit the same wall). `Edit` is hard-blocked on the `.ipynb` extension, and `NotebookEdit` requires a prior successful whole-file `Read` (which fails) and can only replace a whole cell. Both blocks key off the `.ipynb` **extension**, so the standard method is to rename around them:
  1. `mv "<nb>.ipynb" "<nb>.ipynb.txt"` (via the Bash tool).
  2. Edit with the normal loop: `Grep -n` to locate (line numbers work on the JSON text), `Read` with `offset`/`limit` to read slices, `Edit` for exact-match replacement. You are editing **raw notebook JSON**: each source line is a `"…\n",` array element, so preserve escaping (`\"`, `\\`, `\n`) and array structure (commas; the last element of a `source` array has no trailing comma); insert new code lines as new `"…\n",` elements. Copy the escaped form straight from what `Read`/`Grep` shows.
  3. **Always validate before renaming back**: `& "$env:LOCALAPPDATA\miniforge3\envs\inspect_edf\python.exe" -c "import json; json.load(open(r'<nb>.ipynb.txt', encoding='utf-8'))"`. Fix any error before restoring the extension.
  4. `mv "<nb>.ipynb.txt" "<nb>.ipynb"`. Keep the whole rename within a single task so git/Jupyter/Voila always see the normal `.ipynb` path; the diff stays byte-exact except the edited bytes (minimal git diff). This works for both the single-mega-cell Voila files and the multi-cell Jupyter twins, and restores partial reads + locating that the JSON-surgery script cannot offer.
  - **Small notebooks** (whole-file `Read` succeeds, e.g. `1bis_anonymize_edf*` ~26 KB, `4_remap_events_edf*` ~33 KB): keep using the normal `Read` + `NotebookEdit` path — no rename needed.
  - **Fallback — JSON-surgery script**: for changes where editing escaped JSON is genuinely error-prone (many repetitive structural edits, or edits with lots of quotes/backslashes/newlines), prefer a one-shot Python script that `json.load`s the notebook, string-replaces inside the parsed cell `source` (asserting each pattern matches **exactly once**), and `json.dump`s back with `indent=1, ensure_ascii=False` + trailing newline. Operating on the parsed source string avoids all JSON escaping and guarantees valid output by construction; the cost is the per-task script overhead and no in-context reading of the edit site. Do **not** use jupytext/nbconvert round-trips: jupytext is not installed and would clobber cell outputs (10 cells of `2_select&remap_channels_edf.ipynb` carry outputs) and churn cell IDs (large diff).
- **Updating environment.yml**: When adding a new `import` to any tool, add the corresponding package to `environment.yml` as part of the task — conda-forge section if available, pip section otherwise. Do not rely on transitive dependencies for packages that are directly imported.
- **Updating SPEC.md**: At the end of a task, propose updates to SPEC.md for any new feature, workflow change, constant, output file, or module detail — but do not write without explicit confirmation.
- **Updating CLAUDE.md**: If a design decision or development rule changes, update this file directly as part of the task.
- **Read SPEC.md at the start of each new task**: Before implementing anything, read `SPEC.md` once to get an up-to-date picture of the project structure, tool inventory, and planned modules. This avoids re-deriving context that is already documented there.
- **Ask clarifying questions before starting**: Before implementing any non-trivial task, ask the user clarifying questions to resolve ambiguities (format, scope, backward compatibility, etc.), then present a plan for validation before touching any file.
- **Plan-first workflow for new tasks**: For every new non-trivial task, follow this sequence before touching any file: (1) Ask the user as many clarifying questions as needed (expected inputs/outputs, edge cases, UI layout, backward compatibility, interaction with existing tools, output file names, etc.) — do not start planning until ambiguities are resolved; (2) Switch to the most capable available model (currently `claude-opus-4-8`) to draft the plan; (3) Write the plan to a temporary Markdown file (e.g. `tools/plan_<task_name>.md`) so the user can read, annotate, and validate it; (4) Only begin implementation after the user explicitly confirms the plan. The plan file should cover: goal and scope, list of files to create or modify, step-by-step implementation sequence, data flow (inputs → outputs), widgets and UI elements if applicable, error-handling strategy, and any open questions remaining. Delete the plan file once the task is complete and the user confirms.
- **Language in tools and documentation**: All user-facing strings in notebooks (widget labels, print statements, HTML banners, section descriptions) must be in English. SPEC.md must also be written in English.
