# Dream-Toolkit — Inspect_EDF

For full project description, tool details, and planned modules, see [SPEC.md](SPEC.md).

## Key design decisions

- **Custom EDF binary parser for headers**: EDF headers are read using a hand-written binary parser rather than pyedflib. pyedflib was insufficiently robust for the heterogeneous EDF files encountered in real-world datasets (encoding edge cases, non-standard headers). This parser is used **only for header inspection** — it never loads signal data.
- **Sampling frequency = samples_per_record / duration_data_record**: the per-channel 8-byte header field is the *number of samples per data record*, not the sampling frequency. They coincide only when `duration_data_record == 1 s` (classic EDF); EDF+ files often use another record duration (e.g. 0.1 s or 2 s), so the rate must be divided by `duration_data_record` to match MNE (e.g. `40 / 0.1 = 400 Hz`, `512 / 2 = 256 Hz`). The custom parser stores the raw field as `samples_per_record` and derives `sampling_frequency` from it. Keep this in sync across all tools sharing the parser (`1_inspect_edf*`, `2_select&remap_channels_edf*`). See SPEC.md for details.
- **MNE for signal loading**: Whenever actual signal data needs to be loaded (e.g. for spectral analysis or future preprocessing), `mne.io.read_raw_edf()` is used. Do not use pyedflib for signal loading.
- **edfio for EDF export**: When writing EDF files (e.g. in `generate_test_data.py`), use `edfio.EdfSignal()` and `edfio.Edf()` directly — not `mne.export.export_raw()`. Direct edfio usage allows per-channel physical range overrides that MNE's export layer does not expose.
- **In-place header anonymization (`1bis_anonymize_edf*`)**: To anonymize without re-exporting (slow), the anonymizer writes a **new** file (`shutil.copy2`) then overwrites **only** the fixed-width identity fields of the 256-byte general header: `patient_id` (bytes 8–88) and `recording_id` (bytes 88–168). Everything from byte 256 onward (per-channel headers + all signal records) is byte-for-byte identical — verified by `sha256(file[256:])` before/after plus an equal-file-size check. **Originals are never modified**; the user is told to delete them after verifying. `patient_id` → `X X 30-DEC-1899 X_X` (Compumedics anonymized format — strips name *and* birthdate, keeps a placeholder date). `recording_id` keeps the real `Startdate <dd-MMM-yyyy>` token (the recording night is **not** direct PII and downstream sleep tools need it) and blanks only the trailing admin/technician/equipment fields to `X X X`. **`start_date`/`start_time` (bytes 168–184) are left untouched** — verified against real files, this is exactly what Compumedics anonymized exports do (only the birthdate inside `patient_id` is replaced). Companion files (hypnogram `.txt`, `.edf.XML`, `*_event_xml.csv`) are copied + renamed to the new stem with **content unchanged** — verified that Compumedics companions hold no patient name (the `.edf.XML` is a `CMPStudyConfig` of scoring/montage settings); a warning still tells the user to re-check their own export profile.
- **Dual delivery (Jupyter + Voila)**: Every user-facing tool exists in both forms. The Voila version hides all code cells; the Jupyter version is kept in sync for debugging. Always maintain both versions when modifying a tool.
- **TSV + HTML outputs**: TSV files are machine-readable for scripting; HTML reports are human-readable for non-programmers.
- **MNE duplicate channel suffixes**: MNE ≥ 1.8 renames duplicate channel names by appending `-0`, `-1`, etc. Use `drop_suffix_duplicates()` to keep only the `-0` variants, and `adapt_remap_dict_to_suffixes()` to match base channel names in remap dicts to MNE's suffixed names. Apply this pattern whenever loading an EDF with MNE and using a channel remap dict.
- **Physical bounds from MNE internals**: MNE 1.9 stores EDF physical range as `physical_max + offset` in `_raw_extras`, not as explicit `physical_min`/`physical_max`. Use `get_phys_bounds_uV()` (defined in `5_quality_overview_voila.ipynb`, duplicated in `7_live_explore_1file*`) to reconstruct these bounds when needed. **Critical unit gotcha**: MNE keeps `physical_max`/`offsets` in the channel's *native* EDF physical unit (µV, mV, V…), **not** in µV, while `raw.get_data() * 1e6` is always in µV. So `get_phys_bounds_uV()` must multiply both by `extras['units'][ch_idx] * 1e6` (the per-channel native-unit→volts multiplier MNE stores: `1e-6` for µV, `1e-3` for mV, `1.0` for V). Without this, any channel declared in mV (typical for Compumedics **EOG/EMG/ECG**, e.g. `physical_max=1.0 mV`) is compared against a `1.0 µV` bound — 1000× too small — so `bounds_pct` flags ~98–100 % of samples as "at EDF bounds" even on a perfectly healthy signal (EEG channels, declared in µV, were unaffected, which is why the bug stayed latent until non-EEG channels were added). Keep this scaling in sync across all three notebooks. Verified on ICEBERG 117: EOG/EMG/ECG `bounds_pct` 97–98 % → <0.4 %, EEG unchanged.
- **EOG/EMG/ECG channel detection**: classify non-EEG channels by **transducer type OR channel name** (method from `inspect_edf_voila_EMG.ipynb`): EOG = transducer `EOG` or name `eog`; ECG = transducer `ECG`/`EKG` or name `ecg`/`ekg`; EMG = transducer `EMG` or name `emg`/`chin`/`menton` (the `chin|menton` additions cover Compumedics chin-EMG labels). EEG uses `KNOWN_EEG_CHANNEL_RE` (full 10-10 + mastoids + literal `EEG`) or transducer `EEG`/`AGAGCL ELECTRODE`, excluding anything already matched as EOG/ECG/EMG. Reuse `detect_channel_types()` (in `7_live_explore_1file.ipynb`) when a tool needs the scoring montage, and pre-fill it as an editable selection so the user can correct misses.
- **Compumedics event annotations**: the `*_event_xml.csv` companion next to an EDF is already a parsed table (`Name, Start, Duration` in seconds: arousals, apnea/hypopnea, limb movement, SpO2 desaturation…). `7_live_explore_1file.ipynb` overlays events via `load_events(edf_path)` — CSV-first, then a `<ScoredEvents>` parse of the `*.edf.XML` as a fallback when the CSV is absent — returning a `Name/Start/Duration` **DataFrame** plus a `source` tag (`'csv'`/`'xml'`). It mirrors `4_remap_events_edf`'s loader but returns a DataFrame (what the overlay/navigator code consumes) rather than a list of tuples.
- **Configurable event-CSV suffix (`4_remap_events_edf*`)**: the Compumedics event-CSV name is **not** hardcoded — `event_companion_paths(edf_path, csv_suffix='_event_xml.csv')` and `load_events(edf_path, csv_suffix='_event_xml.csv')` take the suffix as an argument (default preserved). Section 1 exposes an editable `CSV suffix:` field with **auto-detection on folder selection**, reusing the hypnogram-suffix detection idiom of `5_quality_overview` (`os.path.normcase`-matched `.csv` files next to each EDF, colour-coded green/orange info line) but selecting the **most frequent** suffix (shortest on ties) rather than the longest — events have no "more specific remapped" variant the way hypnograms do. The chosen suffix is threaded into both the scan and the CSV-vs-XML check; the XML fallback (`.edf.XML`) stays fixed.
- **ipywidgets `Box` default `overflow:auto` (stray per-row scrollbars)**: jupyter-widgets ships `.widget-box { box-sizing: border-box; overflow: auto; }`, so any `HBox`/`VBox` row that carries a `border`+`padding` whose children overflow the content box by even 1–2 px renders a per-row vertical scrollbar (the up/down ▲▼ arrows the user can scroll by accident, shifting the row content). Fix: set `overflow='hidden'` explicitly on such bordered/padded row layouts (applied to the Section 3 "Harmonize labels" rows of `4_remap_events_edf*`). Keep the intended scroll only on the outer list container.
- **Event sourcing & harmonization (`4_remap_events_edf*`)**: scored events are read **CSV-first / XML-fallback** via a shared `load_events(edf_path)` helper — the Compumedics `*_event_xml.csv` (`Name, Start, Duration`) is tried first, then the `<ScoredEvents>` of the `*.edf.XML` (Profusion `CMPStudyConfig`); both verified equivalent on ICEBERG (the tool ships an optional CSV-vs-XML consistency check writing `event_source_mismatch.tsv`). Event *configurations* group files by their `frozenset` of **unique event labels actually present** (not the `<ScoredEventSettings>` profile catalogue). Harmonization produces a single **global** flat JSON `config_param/event_remap.json` (`{raw_label: canonical_label}`, `null` = ignore), merged incrementally across runs with the lenient JSON loader and seeded by `DEFAULT_EVENT_MAPPING` (snake_case canonical vocabulary; arousal subtypes kept, limb laterality collapsed). The XML fallback is also back-ported into `live_explore`'s event overlay (`load_events()`; returns a DataFrame instead of a list of tuples).

- **Event-based epoch rejection (`6_preprocessing_voila`, Section 2 → Epoch rejection)**: the optional 6th rejection method (`event`) flags any 30 s epoch overlapping a **selected canonical** scored-event type. It reuses the shared CSV-first / XML-fallback `load_events(edf, csv_suffix)` and `event_remap.json` (tool 4) — an explicit `event_remap.json` `FileChooser` (auto-pointed at `<edf>/config_param/`) plus an auto-detected, editable `Event CSV suffix` field (most frequent suffix, shortest on ties — same idiom as tool 4). To keep the integration clean, a single ordered registry — `METHOD_ORDER = ['amplitude','flat','gradient','1f_error','1f_r2','event']` with `METHOD_CODE`/`MULTIPLE_CODE` — is the **one source of truth** every helper iterates (`compute_rejection_masks` adds `mask_dict['event']` only when an `event_epoch_mask` is passed; the matrix/heatmap/log/summary/global-table all key off `[m for m in METHOD_ORDER if m in mask_dict]` / `… in df_all['method']`). Consequence: `flag_event` / `event` rows / `*_event` columns appear **only when event rejection actually ran**, so event-free outputs stay byte-compatible with pre-feature runs. The flag is **epoch-level** (replicated across channels → single `flag_event`). Event loading is **non-fatal** per file (no companion ⇒ `⚠` warning, the other 5 methods still run, never added to `failed`); the run-level guard disables it (one warning) when the checkbox is on but no type / no `event_remap.json` is selected. Heatmap colour for `event` is magenta `#e84393` (kept distinct from the green/blue family). A **"Count affected epochs"** button estimates, over the participants checked in Section 3, the epochs each type would flag (overall + per stage) from the hypnogram + events only — **no signal is read**.

- **Custom (non-AASM) sleep stages (`3_remap_hypno*` → `5/6/7`)**: projects may intentionally keep stage labels outside the AASM set (`W/N1/N2/N3/R`), e.g. `N4` or a project movement stage. These are declared **once** in a shared flat JSON `config_param/custom_stages.json` (`{"custom_stages": ["N4", ...]}`, order = display order) **written only by `3_remap_hypno`** and **read** by tools 5/6/7. In tool 3, `current_acceptable()` = `STANDARD_LABELS | {'MT'} | <field>`, so a kept non-AASM target no longer fails the Section 5 verdict; Section 3's "Save remapping" detects non-AASM targets and offers a **➕ Register** button that appends them to the editable `Custom stages` field; Section 4 persists (merges) the declared stages that actually survive in the remapped output into `custom_stages.json`. Tools 5/6/7 each expose an **editable `Custom stages` field auto-filled from the JSON on folder/file selection** — a **volatile per-run override** (it never rewrites the JSON; management stays in tool 3). Three **shared helpers are duplicated** (like `get_phys_bounds_uV`): `load_custom_stages(folder)`, `parse_custom_field(text)`, and `custom_stage_style(custom_stages) -> (stage_y, stage_colors, ytick_pos, ytick_labels)`. **Auto layout/colour**: custom stages stack **below N3** on the hypnogram y-axis (`N3=0 → -1, -2, …` in declaration order); the step-line stays gray with **REM in red** (YASA convention) and each custom stage highlighted in a fixed non-red palette `['#8dd3c7','#ffffb3','#bebada','#80b1d3','#fdb462','#b3de69','#fccde5','#d9d9d9']`. **Hypnospectrogram**: `yasa.plot_spectrogram`/`yasa.Hypnogram` **hard-reject** any non-AASM label (an `assert` on a fixed stage list), so tools 5 & 7 replace it with a **custom `plot_hypnospectrogram()`** that keeps YASA's stage-agnostic spectrogram core (`from yasa.plotting import spectrogram_lspopt`, ~10 lines mirrored) and draws the hypnogram band itself via `custom_stage_style` — no env change (`spectrogram_lspopt` is in the already-required `yasa`). Per-stage outputs include custom stages everywhere: tool 5 `stage_map` gets codes `5+i` and `quality_summary_by_stage`/`dataset_overview` insets iterate `['W','N1','N2','N3','R'] + custom_stages`; tool 6 generates a **dedicated per-custom-stage amplitude-threshold widget** (default 250 µV, rebuilt when the field changes) and the summary/global/heatmap/"Count affected epochs" iterate base+custom; tool 7's `labels_to_yasa_vec`, `validate_hypno` (unknown excludes custom), per-stage PSD, clean-vs-rejected PSD, heatmap and navigator all use `STAGES + CUSTOM_STAGES`. Reading the JSON is **non-fatal** (`[]` on absent/corrupt); an unregistered non-AASM label still falls back to the old behaviour (warning + per-stage exclusion), never a crash.

- **Flat/dead-epoch colour scaling of the hypnospectrogram (`plot_hypnospectrogram()` in tools 5, 7, 7-voila)**: YASA's `plot_spectrogram` (and the in-repo `plot_hypnospectrogram()` that mirrored it) set the colormap range from `np.percentile(Sxx_dB, [trimperc, 100-trimperc])` over **all** pixels. A flat/disconnected 30 s epoch has ~zero power → after the 0.1–40 Hz display filter its dB is a tiny **finite** residual ≈ **−400 dB** (a constant/DC stretch; an exactly-zero stretch goes to `−inf` instead). When the share of such **fully-flat epochs exceeds `trimperc` (2.5 %)**, the lower percentile lands in that floor → `vmin ≈ −400`, the real signal (≈ −20…+15 dB) is crushed into the top ~10 % of the colormap → the plot washes out (uniformly red, with the dead epochs as blue/grey stripes). Note this threshold is on the **fraction of fully-flat *epochs*** (spectrogram columns), **not** the channel's global `flat_pct` (sample-level): a few % of *scattered* flat samples fills no whole epoch and never triggers it; the bug needs **contiguous** flat filling whole epochs (verified on ICEBERG `8_N1`/O1: 16.7 % flat = 130/1189 dead epochs → washout). **Fix** (kept in sync across the 3 copies, like `get_phys_bounds_uV`): convert non-finite `Sxx` to `nan`; per column compute the peak dB `col_max`; a column is **valid** when `col_max > median(col_max) − 60 dB` (60 dB below the median epoch peak — far above normal epoch-to-epoch variation, far above the dead floor, so magnitude-independent); compute `vmin/vmax` from **valid columns only** (fallback to all finite pixels if < 10 valid pixels remain); set invalid columns to `nan` and render them with `cmap.set_bad('#d9d9d9')` (grey = "no signal"). **Clean channels (0 dead epochs) keep every column → byte-identical `vmin/vmax` and image as before.** The tool-7 *navigator* spectrogram (`compute_nav_spectrogram`/`plot_navigator`) is a separate plot that already floors at −120 dB and uses p5–p99 (far less affected) and is **not** changed by this fix. Diagnostic/validation scripts: `tools/simple_hypnospectro_yasa_vs_fix.py` (YASA vs fixed on one real channel) and `tools/compare_flat_spectrogram_fix.{py,ipynb}` (flat-% threshold study on C3 of 8_N1).

## Development context

- **Developer background**: The primary developer is a sleep research engineer with domain expertise in PSG/EEG but limited software engineering experience. Prioritize readability and correctness over abstraction or cleverness.
- **No unnecessary abstraction**: Keep functions explicit and readable. Three clear lines beat a clever one-liner. Do not introduce helper functions or classes unless the complexity clearly justifies it.
- **Error handling in notebooks and new features**: When implementing a new feature or creating a notebook, add `try/except` blocks proactively. Distinguish fatal vs non-fatal failures: a fatal step (e.g. file I/O, epoching) should add the item to a `failed` list and `continue` to the next iteration; a non-fatal step (e.g. re-referencing, report generation) should print a `⚠` warning and continue processing. In Voila notebooks specifically, every button callback and every per-item processing loop must be wrapped so that a single failure never silently crashes the whole run or freezes the UI. Always surface error messages to the user via a widget (e.g. `HTML`, `Output`) or `print()` inside a `widgets.Output` context.
- **Skip existing reports + robust global output updates**: Every processing tool that generates one output file per participant must include a "Skip participants with an existing report" checkbox (checked by default). The pattern has three parts: (1) when the data folder or output folder is selected, an `existing_reports_info` HTML widget displays how many participants already have a report; (2) when "Load participants" is clicked, participants with an existing report are filtered out of the list (not shown), and the info message reports how many were hidden; (3) in the run loop, `attempted_ids` (a `set`) is populated just after the skip check — only participants actually entered into the processing `try:` block are added. Global output files must then be updated using `attempted_ids` as follows: cumulative per-row files (e.g. `global_epoch_rejection.tsv`) are merged with the existing file by removing rows where `file_id ∈ attempted_ids` then concatenating new rows; aggregated summary files (e.g. `global_rejection_by_stage.tsv`) are fully regenerated from all individual per-participant files present in the output folder (not just the current run); `failed.tsv` uses the same merge logic, and entries for participants that previously failed but now succeed are automatically removed (file deleted if it becomes empty). Reference implementations: `5_quality_overview_voila.ipynb` and `6_preprocessing_voila.ipynb`.
- **Comments**: Add them only where the EEG/PSG domain logic would not be obvious to a reader unfamiliar with sleep research (e.g. why a 500 µV threshold flags clipping, or why stage 4 is remapped to N3).
- **No formal test suite**: Validate tools against real EDF files from the dataset. Do not add pytest or unittest infrastructure unless explicitly requested.
- **OS**: Windows 10 with Conda. PowerShell is the default shell. Use forward slashes in Python path strings (MNE and pathlib handle them correctly on Windows).
- **Normalize path comparisons**: whenever two filesystem paths are compared as strings (equality, `in`, `.isin()`, set membership, dict-key lookup), wrap both sides in `os.path.normcase(...)` first. On Windows this removes drive-letter case (`C:` vs `c:`) and `/` vs `\` mismatches (a no-op on POSIX, so it stays cross-platform). Keep the stored/displayed path in its original form; normalize only at the comparison. This prevents "already-processed / skip" checks from silently failing when one path comes from disk and the other from a fresh folder scan. Applied across all current tools (audit complete); see *Cross-cutting procedures → Path comparison normalization* in SPEC.md.
- **Running Python scripts from the terminal**: `conda` is NOT in the PATH. miniforge3 is installed at `$env:LOCALAPPDATA\miniforge3` (not `$env:USERPROFILE\miniforge3`). To run a script, always use the explicit executable path:
  ```powershell
  & "$env:LOCALAPPDATA\miniforge3\envs\inspect_edf\python.exe" tools/my_script.py
  ```
  Do NOT use `conda run`, `conda activate`, or bare `python` — they will all fail.
- **Editing large notebooks (rename-to-`.txt` method)**: The standard notebook tools fail on the larger tool notebooks because `Read` renders **all** cells and ignores `offset`/`limit` on `.ipynb`, so any notebook whose *total* size exceeds the ~25 k-token `Read` cap cannot be read (this is driven by total file size, not by the single-mega-cell structure — the multi-cell Jupyter twins hit the same wall). `Edit` is hard-blocked on the `.ipynb` extension, and `NotebookEdit` requires a prior successful whole-file `Read` (which fails) and can only replace a whole cell. Both blocks key off the `.ipynb` **extension**, so the standard method is to rename around them:
  1. `mv "<nb>.ipynb" "<nb>.ipynb.txt"` (via the Bash tool).
  2. Edit with the normal loop: `Grep -n` to locate (line numbers work on the JSON text), `Read` with `offset`/`limit` to read slices, `Edit` for exact-match replacement. You are editing **raw notebook JSON**: each source line is a `"…\n",` array element, so preserve escaping (`\"`, `\\`, `\n`) and array structure (commas; the last element of a `source` array has no trailing comma); insert new code lines as new `"…\n",` elements. Copy the escaped form straight from what `Read`/`Grep` shows.
  3. **Always validate before renaming back**: `& "$env:LOCALAPPDATA\miniforge3\envs\inspect_edf\python.exe" -c "import json; json.load(open(r'<nb>.ipynb.txt', encoding='utf-8'))"`. Fix any error before restoring the extension.
  4. `mv "<nb>.ipynb.txt" "<nb>.ipynb"`. Keep the whole rename within a single task so git/Jupyter/Voila always see the normal `.ipynb` path; the diff stays byte-exact except the edited bytes (minimal git diff). This works for both the single-mega-cell Voila files and the multi-cell Jupyter twins, and restores partial reads + locating that the JSON-surgery script cannot offer.
  - **Small notebooks** (whole-file `Read` succeeds, e.g. `1bis_anonymize_edf*` ~26 KB, `4_remap_events_edf*` ~33 KB): keep using the normal `Read` + `NotebookEdit` path — no rename needed.
  - **Fallback — JSON-surgery script**: for changes where editing escaped JSON is genuinely error-prone (many repetitive structural edits, or edits with lots of quotes/backslashes/newlines), prefer a one-shot Python script that `json.load`s the notebook, string-replaces inside the parsed cell `source` (asserting each pattern matches **exactly once**), and `json.dump`s back with `indent=1, ensure_ascii=False` + trailing newline. Operating on the parsed source string avoids all JSON escaping and guarantees valid output by construction; the cost is the per-task script overhead and no in-context reading of the edit site. Do **not** use jupytext/nbconvert round-trips: jupytext is not installed and would clobber cell outputs (10 cells of `2_select&remap_channels_edf.ipynb` carry outputs) and churn cell IDs (large diff). **Preserve the original type of `cell["source"]` when writing back, or the git diff explodes.** nbformat allows `source` to be *either* a list of line-strings *or* a single string, and several tool notebooks here (e.g. `3_remap_hypno*`, single mega-cell) store it as **one string** on a single physical JSON line. If you `"".join(...)` to edit and then assign a *list* back, `json.dump(indent=1)` re-serializes that one line into ~1500 lines (huge, unreviewable diff) even though the content barely changed. The fix: detect the original container (`isinstance(cell["source"], str)`) and reassign the **same type** — a single string stays a single string (diff = 1 line changed). **If (and only if) you must write back as a list, split with `src.splitlines(keepends=True)`, never `src.split("\n")`:** the latter drops the trailing `\n` from every element, so nbformat concatenates the lines with no separators and the whole cell collapses onto one physical line — the code then fails to compile (silent corruption that `json.dump` happily writes). Always re-`compile()` the joined source as a syntax gate after writing, precisely to catch this. The rename-to-`.txt` method does not have this pitfall (it edits raw bytes in place and never re-serializes the container), so prefer it when diff minimality matters and the edits aren't too escaping-heavy. One more gotcha: an IDE with the notebook open (VS Code/Jupyter) may **re-serialize it between your tool calls** — normalising a single-string `source` into a list and adding nbformat cell `id`s — so re-read the file before each surgery pass and don't assume your last write is still the on-disk state.
